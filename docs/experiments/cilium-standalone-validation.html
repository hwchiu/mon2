關鍵判讀

Cluster node 路徑

pinned 1.17.6 live lab 在 2026-06-19 已證實純 k8s path 可成立。host firewall 啟用後，server 端與 zone2 端的 host endpoint BPF policy map 都出現了具體 deny entries， cross-zone deny 也和 curl matrix 一致。

Linux deprecated 路徑

external workload registration 與 cluster identity 都存在。更深入的 2026-06-19 診斷也證明， 即使手動重建 VM agent、打開 host firewall，並讓 deny rule 寫進 BPF policy map， 實際打到 Linux host IP 的流量仍沒有命中那些 deny entries。

Windows 分支

Windows VM 與 IIS 都正常，這不是 bootstrap failure。真正缺的是一條可信、官方支援的 standalone Cilium-managed Windows path，因此這一列應歸類為產品限制，而不是 lab error。

主要 live findings

Cluster 端

  • 必須額外啟用 hostFirewall.enabled=true 與 devices=eth0。
  • 必須補上 policy-cidr-match-mode=nodes 才能讓 peer node IP 出現 cidr:* identity。
  • 2026-06-19 重驗證時，server 端 BPF map 已出現對 10.70.10.12/32 的 deny，zone2 端也已對 10.70.10.10/32 與 10.70.10.11/32 寫入 deny。
  • 即使純 k8s path 已通過，live cilium-config 仍維持 enable-node-selector-labels=false；這次通過的是目前 repo 的 node-IP-based host-policy 形狀。

Standalone 端

  • Linux deprecated path 需要先修正 cluster.id = 1 並重建 clustermesh。
  • 診斷時把 Linux VM agent 暫時改成 --enable-host-firewall --devices=eth0 --policy-cidr-match-mode=nodes 後，endpoint 已能進入 policy-enabled: both，而且本地 BPF map 也能看到 deny entry。
  • 但真實測試流量仍經過 Linux host IP 10.70.20.20 放行，deny counter 維持 0 packets；因此這條 deprecated path 不能算可信的 standalone host-IP enforcement。
  • Windows bootstrap template 只安裝 OpenSSH 與 IIS，並沒有安裝 standalone Cilium agent。
  • Windows row 的正確分類是 No official standalone Windows path found，不是 infrastructure bring-up failure。
