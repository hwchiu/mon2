Experiment A: k3s + Cilium + Calico HostEndpoint

This page is a setup validation archive, not a formal identity-based firewall result. It records the earliest compatibility check: whether Calico can be reduced to only HostEndpoint + host firewall while k3s keeps Cilium as the CNI (no takeover of pod networking or IPAM).

架構與範圍

Jumpbox (20.110.199.106 / 10.70.0.10)
   |
   +--> k3s server 10.70.10.10
   |      - k3s control-plane
   |      - Cilium installed
   |      - Calico datastore backed by Kubernetes API
   |
   +--> k3s agent-01 10.70.10.11
   |      - Cilium node
   |      - Calico policy-only active
   |      - manual HostEndpoint on eth0
   |
   +--> k3s agent-02 10.70.10.12
          - Cilium only
          - used as control sample

保留的東西

  • Cilium 仍是唯一 CNI。
  • k3s 節點與 service 網路保持原狀。
  • 所有節點維持 Ready。

加入的東西

  • Calico CRD 與 policy-only 元件。
  • 單一 worker 的 calico-node。
  • 一個手動建立的 HostEndpoint。

刻意沒做的事

  • 沒有讓 Calico 接管 /etc/cni/net.d。
  • 沒有啟用 Calico IPPool。
  • 沒有開啟全叢集 auto HostEndpoints。

叢集側 Calico 設定

已套用

Policy-only overlay

  • 以 upstream calico-policy-only.yaml 為基底。
  • 刪除 install-cni init container。
  • 移除與 host CNI 目錄相關的 volume/volumeMount。
  • 加入 NO_DEFAULT_POOLS=true。
  • 把 calico-node 限縮在 hep-0605-0534-k3s-agent-01。
已套用

Host policy

  • 建立 baseline allow-all policy，selector 為 host_protected == "true"。
  • 建立 hep-0605-0534-k3s-agent-01-eth0 HostEndpoint。
  • HostEndpoint 標籤包含 host_name、host_protected、platform。
  • Host IP 綁定為 10.70.10.11。

已完成測試

測試 方法 結果
CNI 未被接管 檢查 worker 上的 /etc/cni/net.d 通過，只剩 05-cilium.conflist。
Calico IPPool 已停用 套用 NO_DEFAULT_POOLS=true 並刪除 default pool 通過，叢集中沒有 Calico IPPool。
叢集健康檢查 kubectl get nodes + cilium status --brief 通過，3 個節點都維持 Ready，Cilium 為 OK。
HostEndpoint 實際阻擋能力 在 10.70.10.11:18081 起 HTTP server，再套 deny policy 通過，套用 deny 前回應 200，套用後 timeout。
清理後狀態 刪除暫時的 deny policy 通過，叢集保持健康，只保留 baseline allow 與 HostEndpoint。

目前結論

已證明
  • Calico 可以在這個 lab 中被縮成 host policy footprint。
  • 手動 HostEndpoint 可以綁到節點介面並對流量生效。
  • 在單一 worker 節點上，這個模式目前沒有讓 Cilium 或 k3s 失穩。
尚未完成
  • 還沒把同一套模型擴到所有 k3s 節點。
  • 還沒完成 workload-to-VM 與 VM-to-VM 的 label policy 驗證。
  • 還要驗證 standalone VM 的 Felix 能否穩定加入同一組策略平面。
