實驗 B: Standalone VM + Calico Felix

這一頁專門記錄非 Kubernetes 節點的 Azure VM 驗證流程。目標是把一台獨立 VM 納入相同的 Calico HostEndpoint / GlobalNetworkPolicy 模型，同時保持它不是 k3s node、也不把它變成 Kubernetes worker。

目標 VM

主要 VM

hep-0605-0534-legacy-01

10.70.20.20

目前已啟動，預設有 sshd 與 nginx。

預留 VM

hep-0605-0534-legacy-02

10.70.20.21

後續可用於 VM-to-VM 或 label-based allow/deny 測試。

資料來源

Calico 資料平面不在 VM 本機保存，Felix 會透過 kubeconfig 存取 k3s 的 Kubernetes datastore。

部署設計

採用

Felix-only 啟動模式

  • 不在 VM 上啟動 Kubernetes 元件。
  • 不把 VM 註冊成 k3s node。
  • 以 calico-node -felix 當成 systemd 服務。
  • 使用單獨的 service-account kubeconfig 存取叢集 datastore。
  • Ubuntu 22.04 lab 上額外補了 libpcap.so.1 compatibility symlink，讓 extracted binary 可執行。
採用

策略模型

  • 先建立一個只用於 datastore 對應的 Kubernetes Node placeholder。
  • 再建立對應的 Calico Node 物件。
  • 再建立對應的 VM HostEndpoint。
  • 先套 baseline allow，再套暫時性的 deny 測試規則。
  • 在 VM 端額外設定 FELIX_XDPENABLED=false，避免 binary-only 安裝時的 XDP 雜訊。

第一輪驗證結果

測試 方法 結果
VM 可被 Calico 納管 建立 service account kubeconfig、placeholder Kubernetes Node、Calico Node、Felix systemd 通過，Felix 服務為 active，HostEndpoint 狀態顯示 up。
HostEndpoint 維持 baseline allow 套用 VM baseline allow policy 與 HostEndpoint 後，從 jumpbox 存取 10.70.20.20:80 通過，HTTP 回應仍為 200。
Deny policy 生效 套用 vm-experiment-deny-legacy-01-http 後，從 jumpbox 再次存取 :80 通過，請求在 5 秒內 timeout，HTTP code 為 000。
刪除 deny 後恢復 刪除暫時的 deny policy，再從 jumpbox 存取 :80 通過，HTTP 回應恢復為 200。
叢集基線未受影響 檢查 k3s 節點狀態與 cilium status --brief 通過，3 個真實 k3s 節點仍是 Ready，Cilium 為 OK。

目前狀態與後續

現在成立
  • 第一台 standalone VM 已成功納入同一套 Calico host policy 模型。
  • 非 Kubernetes workload 的 host firewall 已可用 label 與 GlobalNetworkPolicy 控制。
  • Felix 目前會持續從 k3s datastore 收到其他節點的 host metadata 更新。
下一步
  • 把相同做法擴到 legacy-02，補上 VM-to-VM 測試。
  • 加入 workload-to-VM 的 selector-based allow/deny 驗證。
  • 評估是否要把 VM 端安裝流程腳本化，避免重複手工設定。
