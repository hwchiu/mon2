Experiment B: Standalone VM + Calico Felix

This page is also a setup validation archive. It records how two non-Kubernetes Azure VMs were brought into the same Calico HostEndpoint / GlobalNetworkPolicy model while remaining outside any k3s cluster and without becoming Kubernetes workers.

目標 VM

baseline VM

hep-0605-0534-legacy-01

10.70.20.20

目前保留 baseline allow profile，扮演 VM-to-VM 測試的來源端。

restricted VM

hep-0605-0534-legacy-02

10.70.20.21

目前已套用 default deny ingress，只開放特定來源與 SSH。

資料來源

Calico 資料平面不在 VM 本機保存，Felix 會透過 kubeconfig 存取 k3s 的 Kubernetes datastore，兩台 VM 共用同一套 label 與 policy 模型。

架構圖與流量路徑

`legacy-01` 與 `legacy-02` 都是獨立 VM，只安裝 `calico-node -felix`。其中 `legacy-02` 使用 restricted profile 驗證 HostEndpoint 的 default deny 與 allow list；`legacy-01` 則保留 baseline profile 當成可控的來源端。

部署設計

採用

Felix-only 啟動模式

  • 不在 VM 上啟動 Kubernetes 元件。
  • 不把 VM 註冊成 k3s node。
  • 以 calico-node -felix 當成 systemd 服務。
  • 使用單獨的 service-account kubeconfig 存取叢集 datastore。
  • Ubuntu 22.04 lab 上額外補了 libpcap.so.1 compatibility symlink，讓 extracted binary 可執行。
  • VM 端標籤模型分成 policy_profile=baseline 與 policy_profile=restricted。
採用

策略模型

  • 先建立一個只用於 datastore 對應的 Kubernetes Node placeholder。
  • placeholder 不放 kubernetes.io/os label，避免 DaemonSet 誤算到 fake node。
  • 再建立對應的 Calico Node 物件。
  • 再建立對應的 VM HostEndpoint。
  • legacy-01 保留 baseline allow；legacy-02 套 ingress default deny 與 allow list。
  • 在 VM 端額外設定 FELIX_XDPENABLED=false，避免 binary-only 安裝時的 XDP 雜訊。

目前驗證結果

測試 方法 結果
VM 可被 Calico 納管 建立 service account kubeconfig、placeholder Kubernetes Node、Calico Node、Felix systemd 通過，Felix 服務為 active，HostEndpoint 狀態顯示 up。
HostEndpoint 維持 baseline allow 套用 VM baseline allow policy 與 HostEndpoint 後，從 jumpbox 存取 10.70.20.20:80 通過，HTTP 回應仍為 200。
Deny policy 生效 套用 vm-experiment-deny-legacy-01-http 後，從 jumpbox 再次存取 :80 通過，請求在 5 秒內 timeout，HTTP code 為 000。
刪除 deny 後恢復 刪除暫時的 deny policy，再從 jumpbox 存取 :80 通過，HTTP 回應恢復為 200。
jumpbox SSH 可維持 在 legacy-02 restricted profile 生效後，透過 jumpbox SSH 到 10.70.20.21 通過，SSH 仍可登入並回傳 hep-0605-0534-legacy-02。
jumpbox HTTP 被 default deny 從 jumpbox 存取 10.70.20.21:80 通過，5 秒內 timeout，HTTP code 為 000。
VM-to-VM allow 生效 從 legacy-01 存取 10.70.20.21:80 通過，HTTP 回應為 200，證明 HostEndpoint source selector 可在 VM-to-VM 路徑上工作。
workload label allow 未命中 access-client (10.42.0.215) 以標籤 access=legacy-http 存取 10.70.20.21:80 未通過預期 allow，實測 timeout；抓包顯示 VM 端看到來源為 10.70.10.12，不是 pod IP。
未標記 workload 被阻擋 blocked-client (10.42.1.232) 存取 10.70.20.21:80 通過，timeout；抓包顯示 VM 端看到來源為 10.70.10.10。
叢集基線與 DaemonSet 狀態 檢查 k3s 節點、Cilium DaemonSet、Calico / Cilium 狀態 通過，3 個真實 k3s 節點仍是 Ready，cilium / cilium-envoy desired 已恢復成 3。

目前狀態與後續

現在成立
  • 兩台 standalone VM 已成功納入同一套 Calico host policy 模型。
  • VM-to-VM 的 label-based allow 在 HostEndpoint 模式下可實際運作。
  • placeholder Node 已做收斂，不再把 Cilium / Envoy DaemonSet desired 拉高。
下一步
  • 若要把 k8s workload label 真正映射到 VM host firewall，需要評估避免 Cilium egress SNAT 的做法，或改成節點 / host label 模型。
  • 可以再補第二組 restricted VM，驗證多個 VM 群組與更多 port-based policy。
  • 仍可把 VM 端安裝流程腳本化，減少手工傳檔與 systemd 安裝步驟。
