Azure Lab / k3s + Cilium / Calico HostEndpoints

Calico HostEndpoint experiment summary

This page documents the current experiment for using Calico only as a host firewall and labeling plane, while keeping Cilium as the Kubernetes CNI. The lab is built on Azure VMs and is designed to answer one narrow question: can Calico HostEndpoints enforce label-driven host policy without taking over pod networking or IP allocation?

Experiment scope k3s with Cilium plus standalone Azure VMs
Calico role Host policy only, no CNI ownership, no IPAM ownership
Current validation One protected k3s worker HostEndpoint is active and tested
VM extension Legacy subnet is provisioned; non-cluster Felix enrollment is next

Architecture

The Azure lab is split into a management subnet, a k3s subnet, and a legacy VM subnet. The k3s cluster provides the Kubernetes datastore for Calico CRDs. Cilium remains the pod networking data plane. Calico is layered in only for host-level policy and HostEndpoint objects.

Management plane
  • Public SSH is limited to the jumpbox.
  • Private access to k3s nodes and legacy VMs goes through the jumpbox.
  • The cluster kubeconfig is tunneled from the local workstation when needed.
Cluster plane
  • k3s runs with Flannel disabled.
  • Cilium owns pod networking and service routing.
  • Calico uses the Kubernetes datastore exposed by the cluster.
Host policy plane
  • Host endpoints are represented as Calico CRDs.
  • GlobalNetworkPolicy selects hosts by labels.
  • Calico IP pools are intentionally disabled.

Calico configuration applied

Applied

Policy-only overlay on the cluster

The cluster-side Calico install starts from the upstream calico-policy-only.yaml manifest and then patches it to reduce blast radius.

  • install-cni init container removed
  • host CNI directory mounts removed from calico-node
  • NO_DEFAULT_POOLS=true added
  • calico-node limited to hep-0605-0534-k3s-agent-01
  • calico-kube-controllers narrowed to the node controller
Applied

HostEndpoint resources

The current cluster exposes Calico resources through the Kubernetes CRD API group crd.projectcalico.org/v1, so the repo manifests were aligned to that API for use with kubectl.

  • Baseline allow-all GlobalNetworkPolicy for host_protected == "true"
  • Manual HostEndpoint on hep-0605-0534-k3s-agent-01 eth0
  • Expected IP pinned to 10.70.10.11
  • No Calico IPPool left in the cluster

What was intentionally not enabled

Not enabled

Calico CNI ownership

Cilium remains the only active CNI. Calico does not write files into /etc/cni/net.d and does not replace the active Cilium config.

Not enabled

Calico IP allocation

The experiment is firewall-focused. Default pools were suppressed and any existing default pool was removed so Calico is not acting as pod or host IPAM.

Not enabled

Cluster-wide auto HostEndpoints

Auto-creation remains disabled. The current validation uses one explicit HostEndpoint on a single worker to keep rollback simple.

HostEndpoint validation tests

The current tests focus on proving three properties: Calico does not take over CNI, the cluster stays healthy after the policy-only install, and a HostEndpoint rule can actually block traffic on a protected host interface.

Test Method Expected result Observed result
CNI ownership check Inspect /etc/cni/net.d on the protected worker after Calico install Only Cilium config remains Passed. 05-cilium.conflist remained the only CNI config.
IPAM suppression Set NO_DEFAULT_POOLS=true, remove default IPPool, re-check cluster state No Calico pool should remain Passed. No IPPool resource remained after cleanup.
Cluster health Check kubectl get nodes and cilium status --brief All nodes Ready, Cilium healthy Passed. All three nodes stayed Ready; Cilium reported OK.
HostEndpoint enforcement Run a temporary HTTP server on 10.70.10.11:18081, then apply a deny policy on that port Traffic should succeed before deny and fail after deny Passed. Pre-policy request returned 200; post-policy request timed out.
Cleanup safety Delete the temporary deny policy after validation Cluster remains healthy with only baseline allow policy left Passed. Temporary deny rule removed; cluster remained healthy.

Current conclusion

Validated

What works now

  • Calico can be reduced to a host-policy-oriented footprint on top of the existing k3s + Cilium cluster.
  • Calico can attach a manual HostEndpoint to a node interface and enforce traffic rules there.
  • The policy-only install can coexist with the current cluster state when narrowed to one worker.
Still open

What remains to test

  • Install Felix on a standalone legacy VM and join it to the same policy model.
  • Verify label-driven policy between Kubernetes workloads and non-cluster VMs.
  • Decide whether the experiment can be widened beyond one k3s worker without destabilizing Cilium.
