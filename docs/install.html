Step 1. Azure Lab 與 k3s 基底

目前的 lab 是用 Terraform 建立 Azure jumpbox、k3s 節點與兩台 legacy VM。 k3s server 啟動時已關閉 Flannel 與內建 network policy controller，讓 Cilium 成為唯一 CNI。 

infra/azure/terraform/
infra/azure/terraform/templates/k3s-server-cloud-init.tftpl
infra/azure/terraform/templates/k3s-agent-cloud-init.tftpl

重點:
- k3s server: --flannel-backend=none
- k3s server: --disable-network-policy
- 安裝完成後由 server 節點安裝 Cilium

Step 2. 在 Cilium 叢集上安裝 Calico Policy-Only

叢集側的 Calico 安裝不是完整 CNI 佈署，而是從官方 calico-policy-only.yaml 出發， 再用本 repo 的 patch 把 CNI 相關動作拿掉，確保它不會寫入新的 CNI config。

使用的檔案

  • manifests/calico/cilium-experiment/kustomization.yaml
  • manifests/calico/cilium-experiment/patch-calico-node.yaml
  • manifests/calico/cilium-experiment/patch-kube-controllers.yaml

本次 patch 目的

  • 移除 install-cni init container。
  • 移除 host CNI 目錄 mount，避免寫入 /etc/cni/net.d。
  • 先把 calico-node 限縮在單一 worker 驗證。
  • 關閉 default pools，避免 Calico IPAM 介入。
kubectl --context=hep-0605-0534-k3s apply -k manifests/calico/cilium-experiment
kubectl --context=hep-0605-0534-k3s apply -f manifests/calico/cilium-experiment/10-host-baseline-allow-all.yaml
kubectl --context=hep-0605-0534-k3s apply -f manifests/calico/cilium-experiment/20-agent-01-hostendpoint.yaml

Step 3. 讓 standalone VM 接到 cluster-1 的 Calico control plane

VM 不加入 k3s，也不跑 kubelet。每台 VM 只安裝 calico-node -felix， 並透過專用 kubeconfig 連回第一個叢集的 Kubernetes datastore。

Cluster 端物件

  • manifests/calico/vm-experiment/00-vm-agent-rbac.yaml
  • 05-legacy-01-k8s-node.yaml / 06-legacy-02-k8s-node.yaml
  • 10-legacy-01-node.yaml / 11-legacy-02-node.yaml
  • 30-legacy-01-hostendpoint.yaml / 35-legacy-02-hostendpoint.yaml

VM 端重點

  • systemd 服務只跑 calico-node -felix。
  • CALICO_NETWORKING_BACKEND=none
  • NO_DEFAULT_POOLS=true
  • FELIX_XDPENABLED=false
FELIX_DATASTORETYPE=kubernetes
KUBECONFIG=/etc/calico/kubeconfig
CALICO_NODENAME=hep-0605-0534-legacy-01
CALICO_IP=10.70.20.20
NO_DEFAULT_POOLS=true
CALICO_NETWORKING_BACKEND=none
FELIX_HEALTHENABLED=true
FELIX_LOGSEVERITYSCREEN=Info
FELIX_XDPENABLED=false

目前兩台 VM 都已完成接入，且 placeholder Kubernetes Node 已拿掉 kubernetes.io/os 標籤， 避免 Cilium DaemonSet 把 fake node 算進 desired replicas。

Step 4. 環境完成定義

目標 結果 狀態
1 個 Kubernetes 叢集 k3s server + 2 agent 均為 Ready 完成
2 台 standalone VM legacy-01、legacy-02 均存在且 SSH 可達 完成
VM 均接入 cluster-1 Calico control plane 兩台 VM 的 calico-node systemd 服務均為 active 完成
Calico 不接管 CNI Cilium 仍是唯一 CNI；Calico 只保留 policy metadata 與 HostEndpoint 路徑 完成

下一階段的 install 目標

下一步會是第二個 Kubernetes 叢集，也就是你提到的第二個 TKS / cluster-2 目標。 這一階段的安裝方向先定義為：Cilium 仍負責 CNI，Calico 只提供 policy/identity 相關元件。

但「由第一個叢集直接作為第二個叢集的集中式 Calico 管理平面」這件事，我在文件裡會先標示為 design target，而不是已驗證完成的能力，避免把目前只對 standalone VM 成立的控制面模型誤寫成多叢集已完成。
