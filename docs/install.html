Step 1. Provision Azure base with Terraform

Terraform creates the jumpbox, cluster-1, two legacy VMs, and optionally a second k3s cluster. cluster2_enabled was turned on for this run, creating the 10.70.30.0/24 subnet for cluster-2. 

infra/azure/terraform/

Key variables:
- cluster2_enabled = true
- cluster2_agent_count = 2
- cluster2_subnet_cidr = "10.70.30.0/24"
- cluster2_cluster_cidr = "10.52.0.0/16"
- cluster2_service_cidr = "10.53.0.0/16"

Step 2. Make k3s use Cilium as the only CNI

Both k3s server cloud-init templates disable Flannel and the built-in network policy controller, then install Cilium on the server node. Pod networking, service routing and datapath remain owned by Cilium. 

infra/azure/terraform/templates/k3s-server-cloud-init.tftpl
infra/azure/terraform/templates/k3s-agent-cloud-init.tftpl

Key flags:
- --flannel-backend=none
- --disable-network-policy
- cilium install --wait

Step 3. Install Calico policy-only on cluster-1's Cilium

We are not replacing the CNI. We keep only the pieces required for HostEndpoint, GlobalNetworkPolicy, kube-controllers, Typha and Felix, while removing anything that would take over CNI or write CNI config.

Manifests used

  • manifests/calico/cilium-experiment/kustomization.yaml
  • manifests/calico/cilium-experiment/patch-calico-node.yaml
  • manifests/calico/cilium-experiment/patch-kube-controllers.yaml
  • manifests/calico/cilium-experiment/10-host-baseline-allow-all.yaml

What the patches do

  • Remove the install-cni init container.
  • Remove host CNI dir mounts (avoid writing /etc/cni/net.d).
  • Keep Calico CRDs and the host policy path.
  • Set NO_DEFAULT_POOLS so Calico IPAM stays out.
kubectl --context=hep-0605-0534-k3s apply -k manifests/calico/cilium-experiment
kubectl --context=hep-0605-0534-k3s apply -f manifests/calico/cilium-experiment/10-host-baseline-allow-all.yaml

Step 4. Onboard standalone VMs

The two legacy VMs do not run kubelet and are not cluster nodes. They only run calico-node -felix and reach cluster-1 using a dedicated datastore-only kubeconfig.

Cluster-1 side objects

  • manifests/calico/vm-experiment/00-vm-agent-rbac.yaml
  • 05-legacy-01-k8s-node.yaml / 06-legacy-02-k8s-node.yaml
  • 10-legacy-01-node.yaml / 11-legacy-02-node.yaml
  • 30-legacy-01-hostendpoint.yaml / 35-legacy-02-hostendpoint.yaml

VM environment

  • FELIX_DATASTORETYPE=kubernetes
  • KUBECONFIG=/etc/calico/kubeconfig
  • CALICO_NETWORKING_BACKEND=none
  • NO_DEFAULT_POOLS=true
  • FELIX_XDPENABLED=false

Step 5. Create the second k3s cluster

Cluster-2 is a normal k3s + Cilium cluster. The point is not to install another Calico inside it, but to have a clean independent Cilium cluster first so its own kube-api and datapath are solid. 

terraform -chdir=infra/azure/terraform plan \
  -var='cluster2_enabled=true' \
  -var='cluster2_agent_count=2' \
  -out=tfplan-cluster2

terraform -chdir=infra/azure/terraform apply tfplan-cluster2

Important outputs:
- cluster2_server_private_ip = 10.70.30.10
- cluster2_agent_private_ips = 10.70.30.11, 10.70.30.12
- ssh_cluster2_server_via_jumpbox_command
- ssh_cluster2_kubeapi_tunnel_command

Step 6. Join cluster-2 nodes as external hosts to cluster-1

This was the core addition for this iteration. Inside cluster-2 we do not install the Calico operator or CNI. On every node we additionally run a Felix-only calico-node systemd service that points at cluster-1's datastore.

Automation assets

  • scripts/onboard-external-k8s-hosts.sh
  • manifests/calico/external-k8s-hosts/README.md
  • manifests/calico/vm-experiment/20-host-baseline-allow-all.yaml

What the script creates

  • Placeholder Kubernetes Node objects in cluster-1
  • Corresponding Calico Node objects
  • Corresponding HostEndpoint objects
  • calico-node -felix service on each cluster-2 node
./scripts/onboard-external-k8s-hosts.sh \
  --cluster1-kubeconfig /home/ubuntu/.kube/hep-0605-0534-k3s.yaml \
  --cluster2-kubeconfig /home/ubuntu/.kube/hep-0605-0534-k3s2.yaml \
  --cluster1-api-server https://10.70.10.10:6443 \
  --jumpbox azureuser@20.110.199.106 \
  --cluster-id hep-0605-0534-k3s2

The script was fixed to read binaries and kubeconfig relative to its own location instead of depending on $HOME after sudo, so Felix deployment to the three cluster-2 nodes is now stable.
